## Solution Design

### Fix 1: `POST /api/messages` rejects self-message (HTTP 400)

In `src/server/http/im-routes.ts` â†’ `IMRoutes.sendMessage`:
- After resolving `to` to canonical address, check if the UUID part of `to` matches the authenticated sender's `requester.agentId`
- On match â†’ return **400 SELF_MESSAGE_NOT_ALLOWED**

```typescript
// new check after `const to = ...`
const toUuid = to.split('@')[0];
const fromUuid = requesterAgentId ?? from.split('@')[0];
if (toUuid && fromUuid && toUuid === fromUuid) {
  return Response.json(
    { error: 'Cannot send to self', code: 'SELF_MESSAGE_NOT_ALLOWED' },
    { status: 400 }
  );
}
```

### Fix 2: `POST /api/messages` rejects unregistered recipient (HTTP 404)

In `src/server/http/im-routes.ts` â†’ `IMRoutes.sendMessage`:
- After canonical resolution, attempt to resolve `to` via `resolveIdentifier` (wired by server to `findAgentByEmailOrAddress`)
- On miss â†’ return **404 RECIPIENT_NOT_FOUND**

```typescript
// requires resolveIdentifier wired (server wires it)
if (this.resolveIdentifier && !this.resolveCanonicalAddress(toRaw)) {
  return Response.json(
    { error: `Recipient email not registered: ${toRaw}`, code: 'RECIPIENT_NOT_FOUND' },
    { status: 404 }
  );
}
```

### Fix 3: WS push defense-in-depth (skip self-echo in pushMessage)

In `src/server/http/index.ts` â†’ `BountyHTTPServer.pushMessage`:
- Compare `message.from` UUID part against the target connection's `address` UUID part
- If equal â†’ skip the push (return `false` indicating "not online" semantically, or `true` with no-op)

```typescript
pushMessage(address: string, message: Message): boolean {
  const client = this.clients.get(address);
  if (!client) return false;

  // v0.14.2: Defense-in-depth: skip push when the connection address's
  // UUID matches the message sender's UUID. The route layer (POST /api/messages)
  // rejects self-message at HTTP 400, but this catches the legacy WS message
  // event path (handleWsMessage case 'message') which doesn't go through IMRoutes.
  const connectionUuid = (client.address || '').split('@')[0];
  const fromUuid = (message.from || '').split('@')[0];
  if (connectionUuid && fromUuid && connectionUuid === fromUuid) {
    return false;  // skip self-push
  }
  // ... existing push logic
}
```

### Fix 4: WS push payload consistency

The existing `pushMessage` already enriches with `fromEmail` / `toEmail` via `resolvePushEmail`. The `enrichedData` object is unchanged â€” we only add an early-return path before constructing it.

## Risks + Compatibility Notes

### Risk 1: CLI `--to-email <self-email>` was previously working
The `bounty com send -F A -T A` scenario currently works (sender can message self). After this fix it returns 400.
This is intentional â€” user explicitly chose Plan C and acknowledged this UX change.

### Risk 2: Old agents sending to unregistered addresses
Pre-v0.14.2: the message was stored with `to_address = toRaw` (the raw email) and never pushed. After this fix: HTTP 404 returned.
This is intentional â€” surfaces the wrong-address problem earlier.

### Risk 3: WS reconnect behavior
The WS reconnect on `handleWsOpen` re-sends pending messages. The self-echo skip in `pushMessage` only affects the *push* path; pending messages on reconnect go through `socket.send` directly without the `pushMessage` call. We need to verify this doesn't leak self-messages on reconnect â€” covered by T4 (initial self-send) and integration tests.

### Risk 4: WS message event path (handleWsMessage)
T4 specifically tests this path because it doesn't go through `IMRoutes.sendMessage`. The defense-in-depth fix in `pushMessage` catches it via the same UUID comparison.
